If `v0.6.1` was the release where alchemy could deploy
anything, the next stretch was the release where alchemy
could deploy *the kind of thing people actually wanted to
deploy*: a web app on Cloudflare, with a typed binding
graph, a real CI/CD story, and remote state that didn't
require you to run your own S3 bucket.

It also crosses the 2.0 boundary. The break wasn't
arbitrary ‚Äî Effect 4's beta train (`effect@4.0.0-beta.48`)
broke source compatibility, and we used the forced major
to also unify the resource API surface across providers.
Everything below `v2.0.0-beta.1` is the v0 line tightening
its DX; everything above is the rewrite.

## Web apps come online (`v0.7.0` ‚Üí `v0.11.0`)

### `v0.7.0` ‚Äî Vite, `Output.as<T>()`, the Artifact service

The release that made "deploy a Vite app to Cloudflare"
a one-liner. **John Royal**
([#41](https://github.com/alchemy-run/alchemy-effect/pull/41)) landed
`Cloudflare.Vite` ‚Äî a wrapper that runs `vite build`
through the same bundler pipeline as `Cloudflare.Worker`,
emits the SSR Worker + static assets in one pass, and
attaches both to a single deployment.

```typescript
const site = yield* Cloudflare.Vite("Site", {
  main: "./src/server.ts",
  bindings: { CACHE: kv },
});
return { url: site.url.as<string>() };
```

Two engine additions made it work:

- **`Output.as<T>()`** ‚Äî narrow an `Output<unknown>` to
  the type you know it carries, without coercion. Mostly
  needed at the stack-output boundary, where Effect's
  variance rules don't help.
- **`Artifact` service** ‚Äî a build-artifact cache that
  survives between `alchemy plan` and `alchemy deploy`,
  so a `vite build` doesn't re-run if nothing in the
  source tree has changed since the plan.

### `v0.7.1` ‚Äî `--force`

`alchemy deploy --force` ignores the plan's "no changes"
short-circuit and re-applies every resource. Useful when
something in the cloud has drifted out from under us and
we want the engine to converge it back.

### `v0.8.0` ‚Äî async bindings, take one

The first cut of "async-alchemy"-style bindings:
`R2Bucket.bind` returns a typed callable wrapped in an
Effect, so the binding setup phase can do work (resolve
the bucket name, fetch credentials, set up the client)
before the runtime call begins. This is the shape that
every Cloudflare binding has followed since.

### `v0.9.0` ‚Äî bundle stability, parallel caching

`v0.9.0` is mostly under-the-hood: precreate workers with
tags before resolving DO namespace IDs (so a fresh worker
+ DO deploy doesn't fail on "namespace not found");
retry eventually-consistent `ContainerApplicationNotFound`;
parallel caching via `Deferred`; bundle reports via Sonda;
recover from partial `ContainerApplication` failures.

Plus the Bun-side test helpers (`Test/Bun`) ‚Äî the
counterpart to the Vitest helpers that already existed ‚Äî
landed here. Bun got primary-citizen status as a test
runtime; the `bun test` workflow has been the default
since.

### `v0.9.1` ‚Äî silently-dropped env vars

A real-world bug caught by **cooper**
([#49](https://github.com/alchemy-run/alchemy-effect/pull/49)):
`Cloudflare.Worker` `env` props were being typed as
optional and then silently dropped in the
serialization path. Fixed.

### `v0.10.0` ‚Äî `Output<T>` in Worker env + Vite props

**cooper** again
([#51](https://github.com/alchemy-run/alchemy-effect/pull/51)): a
`Worker`'s `env` and a `Vite`'s `env` now accept
`Output<T>` directly, not just resolved strings. This is
what lets you wire a backend Worker's URL into a Vite
SPA's `VITE_*` at deploy time without any string
hackery.

```typescript
const api = yield* Cloudflare.Worker("Api", { ... });
const web = yield* Cloudflare.Vite("Web", {
  env: { VITE_API_URL: api.url.as<string>() },
});
```

`DynamicWorker` was also renamed to
`DynamicWorkerLoader` here ‚Äî the old name was misleading
(it was the *loader*, not the loaded worker).

### `v0.11.0` ‚Äî tree-shakable providers

**The migration to `Provider.effect`** ([#53](https://github.com/alchemy-run/alchemy-effect/pull/53)).
Every provider was registered up-front in a single
`AllProviders` Layer, which meant importing one S3
bucket dragged the entire AWS provider catalogue into
your bundle. The migration to per-provider
`Layer.effect(Tag, ...)` registration made each provider
independently importable ‚Äî your bundle now only
contains the providers you actually use.

Plus: KV per-operation bindings (`KVNamespace.get`,
`.put`) removed in favor of the unified
`KVNamespace.bind(...)`. The per-op bindings were a
holdover from before the `Binding.Service` pattern
existed; with it, they were redundant.

## The 2.0 rewrite begins (`v2.0.0-beta.1` ‚Üí `beta.9`)

### `beta.1` ‚Äî Effect 4 + `Alchemy.Stack`

The release notes for `beta.1` are short by design: it's
the version-bump release. But what the bump *forced* was
two things:

- **Migration to `effect@4.0.0-beta.48`.** Effect 4's
  Schema overhaul and runtime restructure broke source
  compatibility for every provider. Cleanup pass.
- **`Alchemy.Stack`** as the canonical stack constructor
  ‚Äî the typed handle that exports the stack outputs,
  takes `providers` and `state` as Layers, and serves as
  the cross-stack reference target you `yield* Backend`
  against (the latter use case didn't ship until
  `beta.37`, but the shape was laid down here).

Plus: `R2Bucket.bind(resource)` works on *every*
Cloudflare resource (Worker, Vite, Workflow, DO, ‚Ä¶) ‚Äî
binding setup is no longer per-resource-type. And R2
`put` now requires `Content-Length`, which fixed a class
of edge cases where unbounded `ReadableStream` inputs
were being silently rewritten.

### `beta.2` ‚Äî ProviderCollections

`ProviderCollections` collapses N provider Layer types
into one. Before this, `R` requirements on a stack effect
looked like a 40-line union; after, it's
`Cloudflare.Providers | AWS.Providers`. The wire shape
didn't change ‚Äî the public collections are just a
named alias for `Layer.mergeAll(...)` of every provider
in a cloud.

### `beta.3` ‚Äî GitHub `Comment`, `Secret`, `Variable`

The first GitHub resources. `Comment` posts a comment on
a PR (great for `alchemy plan` CI output); `Secret` and
`Variable` write to repo/org-level GitHub Actions
secrets. The CI guide
([Guides ‚Ä∫ CI](/guides/ci)) walks through a real
workflow.

### `beta.4` ‚Äî async DO + Assets, custom domains, `AuthProvider`

A grab bag with a strong DX thread:

- **Async `DurableObjectNamespace` binding** ‚Äî same
  pattern as R2, but for DOs: `DurableObjectNamespace.bind(Counter)`
  returns the typed namespace with `getByName(id)` available
  in the binding setup.
- **Async Assets binding** for `Worker`'s asset bindings.
- **`Cloudflare.WorkerCustomDomain`** ‚Äî attach a custom
  hostname to a Worker without going through
  `Cloudflare.Zone` + `Cloudflare.DnsRecord` yourself.
- **Alarms + ScheduledEvents** on DOs (the
  one-shot/scheduled handler counterparts to fetch).
- **`AuthProvider`** ([#61](https://github.com/alchemy-run/alchemy-effect/pull/61))
  ‚Äî generic plumbing for cloud-provider authentication
  that the per-cloud auth implementations (`Cloudflare`,
  `GitHub`, AWS profiles) all share. Made
  `alchemy login` possible in `beta.11`.
- **CLI binary renamed** from `alchemy-effect` to
  `alchemy` (long overdue).
- **`platform-bun` / `platform-node` split.** Resolved at
  runtime ‚Äî Bun-native services where available, Node
  fallback otherwise.

### `beta.5` ‚Äì `beta.9` ‚Äî packaging shake-out

These five releases are mostly the consequences of
shipping `beta.4`: the bundled `alchemy` binary worked,
`bun alchemy ‚Ä¶` works, the `cloudflare` workers run with
`nodejs_compat` by default, compatibility flags are
consistent across worker variants, the
`@alchemy.run/stack` export path exists.

`beta.9` ([#66](https://github.com/alchemy-run/alchemy-effect/pull/66))
unbundles the CLI into its own package
(`@alchemy.run/cli`), and **Michael K** sets up the
`@alchemy.run/*` package layout that the rest of 2.0
operates inside.

## Cloudflare gets serious (`beta.10` ‚Üí `beta.15`)

### `beta.10` ‚Äî container readiness backoff

**Christopher Yovanovitch** + **Sisyphus**
([#74](https://github.com/alchemy-run/alchemy-effect/pull/74),
[#76](https://github.com/alchemy-run/alchemy-effect/pull/76))
cut the Container readiness retry schedule from 104s to
30s, and stopped discarding the result of `monitor()` ‚Äî
two long-standing flakiness sources in the
Container-DO smoke tests.

### `beta.11` ‚Äî Queues, `alchemy login`, Vite 7

The headline of `beta.11` is **Cloudflare Queues**,
contributed by **Christopher Yovanovitch** + **Sam**
([#69](https://github.com/alchemy-run/alchemy-effect/pull/69)):
`Queue`, `QueueBinding`, `QueueConsumer` as three
discrete resources, with the consumer side wired as a
binding on a Worker. The Effect-native `Stream`-shaped
consumer API (`Cloudflare.messages(queue).subscribe(...)`)
that `beta.35` shipped is built on top of this surface.

The **`alchemy login`** CLI command ([#67](https://github.com/alchemy-run/alchemy-effect/pull/67))
landed too ‚Äî OAuth-style flow for Cloudflare, AWS SSO,
GitHub. The credentials are stored under a profile, so
you can have separate work / personal logins
side-by-side and select between them with
`--profile`.

Plus Vite 7 support ([#68](https://github.com/alchemy-run/alchemy-effect/pull/68))
and `Cloudflare.Container` learning how to install
external deps in its generated Dockerfile
([#75](https://github.com/alchemy-run/alchemy-effect/pull/75)).

### `beta.12` ‚Äî SecretsStore + account-regional asset buckets

**Michael K**'s **`Cloudflare.SecretsStore`**
([#64](https://github.com/alchemy-run/alchemy-effect/pull/64)) ‚Äî a
real, Cloudflare-hosted secrets store as a resource
that you can `bind` to a Worker. Up to this release,
"secrets in Workers" meant either committing them
encrypted in the alchemy state or shipping them as
plaintext `vars`. SecretsStore is the third option,
backed by Cloudflare's native secret API.

On the AWS side: asset buckets for `StaticSite` /
`SsrSite` now use the account-regional naming pattern
(`alchemy-assets-${accountId}-${region}`), avoiding the
S3 global-namespace collisions that previous deploys
were running into.

### `beta.13` ‚Äî CLI ships correct platform versions

A small but load-bearing fix from **Michael K**
([#91](https://github.com/alchemy-run/alchemy-effect/pull/91)).

### `beta.14` ‚Äî the Cloudflare State Store

**Sam + Michael (Pear)**
([#94](https://github.com/alchemy-run/alchemy-effect/pull/94)).
The Cloudflare-backed **remote state store** ‚Äî alchemy's
own state stored in a Cloudflare Worker + KV namespace,
deployed automatically the first time you run `alchemy
plan`. No more "where do I put the state file?" ‚Äî
remote state by default, encrypted at rest, queryable
from anywhere.

The **`alchemy state`** CLI command also lands here:
`alchemy state ls`, `alchemy state get <key>`, and the
`tree` subcommand for navigating nested scopes.

[Concepts ‚Ä∫ State store](/concepts/state-store).

### `beta.15` ‚Äî `UserApiToken`, `AccountApiToken`, Redacted everywhere

**`UserApiToken`** and **`AccountApiToken`** are
Cloudflare-side IAM resources ‚Äî create a scoped token in
the same stack that needs it, instead of provisioning
it by hand in the dashboard.

`Redacted<T>` (Effect's secret-wrapping type) is now
threaded through the `Plan` JSON, the state store, and
the CLI output ‚Äî secrets stay redacted end-to-end
unless explicitly unwrapped by code that owns them.

## Auth, state, secrets mature (`beta.16` ‚Üí `beta.20`)

### `beta.16` ‚Äî Artifacts Store + GitHub auth provider

**`Cloudflare.ArtifactsStore`** ‚Äî the Cloudflare-backed
counterpart to the v0.7 `Artifact` service. Build outputs
get a stable cache key derived from the bundler input
and are stored in Cloudflare R2; subsequent deploys with
the same key skip the build entirely.

**`GitHub.AuthProvider`** ‚Äî fall through `GH_TOKEN` env
‚Üí a configured PAT ‚Üí `gh auth status` from the CLI.
The same shape `Cloudflare.AuthProvider` uses; the same
shape AWS profiles use.

Plus **`alchemy profile show`** ‚Äî print the active
profile and which auth source it resolved through.

### `beta.17` ‚Äî small CLI polish, ECS `CapacityProvider`

CLI: `alchemy profile clear`, `alchemy state clear`,
don't prompt for approval when the plan has no changes,
don't print `undefined` outputs at the end of an apply.

**ECS `CapacityProvider`** ([#103](https://github.com/alchemy-run/alchemy-effect/pull/103))
from **Jordan**.

### `beta.18` ‚Äî CloudFront, GitHub `Secrets` helper

**CloudFront** picks up `CachePolicy`,
`OriginRequestPolicy`, `ResponseHeadersPolicy`,
`PublicKey`, and `KeyGroup` ‚Äî the cache configuration
surface that AWS-side static-site delivery actually
needs. Contributed by **Jordan** + **Claude Opus 4.7**
([#106](https://github.com/alchemy-run/alchemy-effect/pull/106)),
one of the cleanest AI-assisted PRs in the repo: the
Pulumi/Terraform reference materials drove the resource
shape; Jordan reviewed and integrated.

`GitHub.Secrets` ‚Äî a multi-secret helper that wraps N
`GitHub.Secret` calls into one resource, deduplicating
across repos.

### `beta.19` ‚Äî Axiom resources, tree-shaking via pure annotations

**Axiom** resources ‚Äî `Dataset`, `Monitor`, `Dashboard`,
and a typed `Chart` builder. The dashboards have full
schema validation: if the chart references a field that
isn't in the dataset schema, the plan fails before it
calls the Axiom API.

**Pure-call annotations.** The engine now annotates
provider constructors with `/* @__PURE__ */` so tree
shakers can drop providers that nobody references. The
auto-detect for the entry package is built into the
bundler, so library consumers get this for free.

### `beta.20` ‚Äî Effect `beta.58` + distilled `0.13`

A pure infrastructure update. Effect 4 ticked from
`beta.48` to `beta.58`; `@distilled.cloud/*` from `0.12`
to `0.13` (mostly schema updates). No user-visible
changes, but downstream beta.20 is what every later
provider authored against.
