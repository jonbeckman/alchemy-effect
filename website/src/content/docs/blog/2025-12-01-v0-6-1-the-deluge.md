## Why this release was so big

Two things converged.

**1. The migration to `@distilled.cloud`.** Up to this
point each AWS resource had been calling either
hand-rolled HTTP clients or `aws-sdk-js-v3` directly.
[`@distilled.cloud`](https://distilled.cloud) is a set of
generated, schema-driven AWS SDK packages that ship the
operation contracts (request/response shapes, error
codes, retry classes) as plain TypeScript â€” no runtime
service catalogue, no v3 middleware chain. Switching to
distilled made it tractable to add a service per day.

**2. AI-driven provider scaffolding.** Once the AWS SDK
contracts were typed and queryable, *generating* the
resource provider for a new service became something
Claude could one-shot from the `processes/AWS.md`
playbook. The first AI-co-authored providers in alchemy
ship in this release â€” [DynamoDB Bindings](https://github.com/alchemy-run/alchemy-effect/commit/cdfd45b)
and [Kinesis Stream](https://github.com/alchemy-run/alchemy-effect/pull/30) â€” and
the cadence didn't slow down after.

The combination was the lever. Below is what came out
the other side.

## The AWS services that landed

A non-exhaustive map, by service:

### Compute and orchestration

- **Lambda.** `consumeTable` for DynamoDB stream

## The Cloudflare compute story

The AWS expansion got the headlines but the Cloudflare
work in this release was equally significant: it added
everything you need to run a serverful app on Workers.

**Durable Objects + Containers** ([#37](https://github.com/alchemy-run/alchemy-effect/pull/37)).
The first end-to-end shape of `DurableObjectNamespace`
as a bindable resource â€” DO classes are declared as
nested `Cloudflare.DurableObject<â€¦>()` classes, mounted
on a `Worker`, and reachable via a typed `getByName(id)`
fetcher. Container DOs land in the same change: build a
Dockerfile, push to Cloudflare, expose it over the same
namespace shape.

```typescript
class Counter extends Cloudflare.DurableObject<Counter>()(
  "Counter",
  Effect.gen(function* () {
    return { increment: Effect.fn(function* () { /* ... */ }) };
  }),
) {}

export default class Api extends Cloudflare.Worker<Api>()(
  "Api",
  { main: import.meta.filename },
  Effect.gen(function* () {
    const counters = yield* Counter;
    return { fetch: /* call counters.getByName(id).increment() */ };
  }),
) {}
```

**Workflows.** Cloudflare's durable execution primitive,
exposed as a `Cloudflare.Workflow<â€¦>()` class with the
same shape. (Typed I/O didn't arrive until `beta.37` â€”
in `v0.6.1` workflows were `unknown`-typed in both
directions.)

**DynamicWorker.** A Worker that can spawn other Workers
at runtime â€” `env.LOADER.get(name).run(...)`. Renamed to
`DynamicWorkerLoader` in `v0.10.0`.

**`HttpServer` for Lambda and Cloudflare.** A shared
adapter that lets the same `HttpEffect` server run on
either platform. The Worker side became the foundation
for the schema-driven HTTP API / RPC pattern that
`beta.41` finally stabilized.

## The engine work that made it possible

A lot of the resource breadth in this release depended
on engine changes that landed quietly alongside:

- **Circular binding cycles.** Two resources binding to
  each other (Worker â†’ Worker, Lambda â†’ KMS key â†’
  Lambda) plan as a graph with cycles. The two-phase
  apply was introduced here; the conceptual write-up
  later became
  [Circular references, without the deadlock](/blog/2026-04-25-circular-references).
- **`Construct.fn` and two-phase apply.** Generalize the
  "reserve identifiers in pass 1, finish reconciling in
  pass 2" pattern so any provider could opt in.
- **Replace on top of a partially-replaced resource.**
  If a previous apply crashed halfway through a
  replacement, the next plan picks up where it left
  off instead of refusing to proceed.
- **`Stack.make` capturing providers.** Resource providers
  used to be implicit in the surrounding `Layer`; this
  change made them part of the stack's typed signature,
  enabling the later `ProviderCollections` refactor in
  `beta.2`.

## The bundler and CLI

- **Generalized bundler.** End-to-end deployment of a
  Lambda or Worker now goes through the same bundler
  pipeline â€” typed entry, shared tree-shaking, shared
  `.alchemy/tmp` output. Builds across services compose
  instead of fighting each other.
- **Docker commands** for building Container images
  reproducibly from the same bundler output.
- **CLI: `tail` and `logs`.** Filter by resource ID;
  read from CloudWatch Logs for AWS or the platform
  Tail API for Cloudflare. Same UX.
- **`Process` module + daemon.** A long-running process
  manager so `alchemy dev` (later) could keep sidecars
  alive across plan iterations.

## Kubernetes, finally

A first-cut Kubernetes provider â€” `Manifest`, `Service`,
`ServiceAccount`, and the usual core APIs â€” landed in
the same release. It's not a feature-flagship for
alchemy (we've leaned harder into serverless since), but
it proved the engine wasn't AWS-specific. If you can
write a typed CRUD lifecycle, alchemy can deploy it.

## What didn't ship until later

For completeness, things that *looked* like they should
be in v0.6.1 but didn't make it:

- The `reconcile` lifecycle hook (`create` + `update`
  unified into one observe-and-converge function) â€” that
  came in `beta.31`.
- `--adopt` for taking ownership of pre-existing cloud
  resources â€” `beta.24`.
- Dev mode for Cloudflare Workers â€” `beta.24`.
- Cloudflare-backed remote state â€” `beta.14`.
- Neon, Hyperdrive, AiGateway, Tunnel/VpcService â€”
  spread across `beta.29` â†’ `beta.31`.

But by the time `v0.6.1` was tagged, the project had
crossed a line: there was enough breadth that *somebody
else* could deploy a real app with it. The next few
months were mostly about closing the gaps that emerged
once people did.
