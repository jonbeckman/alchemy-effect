The first six minor releases of `alchemy-effect` are the
ones where the engine stopped being a sketch and started
deploying things. By the end of `v0.6.0` there was a
working `Plan`/`Apply` engine on top of Effect, a CLI
that could run a stack, two AWS resources (DynamoDB Table,
VPC), four Cloudflare resources (Worker, Assets, R2, KV),
and enough of EC2 networking — Internet Gateway, Route,
Route Table, NAT Gateway, EIP, Security Group — to put a
real Lambda in a real VPC.

## v0.1.0 — The engine takes shape

Alchemy started as the answer to a single question: *can
the lifecycle of a cloud resource be expressed as a typed
Effect?* The early `v0.1.0` commits were almost entirely
about types — the engine itself worked, but TypeScript's
`NoInfer` and `Extract` limitations made the API painful
in surprising places.

A handful of structural decisions landed in this release
and have survived every subsequent rewrite:

- **Currying everywhere.** A `Resource("AWS.S3.Bucket")`
  call returns a builder; the second call attaches the
  provider. The currying split was forced by `NoInfer` /
  `Extract` — you can't infer `Props` and `Attributes`
  from the same call site — but it also gave us a
  natural place to declare the type identity of a
  resource separately from its lifecycle code.
- **HKTs removed from Capability and Resource.** Earlier
  drafts used higher-kinded types to keep input/output
  shapes generic; in practice TypeScript collapsed them
  too aggressively and error messages were unreadable.
  Replacing them with plain generic parameters made the
  whole system tractable.
- **`BindingTag`** — a tag type that maps a `Binding`
  declaration to the `Binding.Service` runtime
  implementation. This is what lets `R2Bucket.bind(...)`
  and `Lambda.InvokeFunction.bind(...)` share one
  uniform syntax across providers.
- **`Provider.effect`** as the canonical way to wire a
  Layer into a provider. Every resource provider since
  has been an `Effect.fn(function* ({ id, news, output })
  { ... })` plugged into this builder.

There were no end-user resources yet — the release was
just the type machinery, the plan/apply scaffolding, and
the planner threading bindings through state.

## v0.2.0 — First resources, first tests

`v0.2.0` is where the engine actually deployed something.
**John Royal**'s [#13](https://github.com/alchemy-run/alchemy-effect/pull/13)
landed the four foundational Cloudflare resources in one
go: `Worker`, `Assets`, `R2Bucket`, `KVNamespace`. The
AWS side picked up `DynamoDB.Table` ([#10](https://github.com/alchemy-run/alchemy-effect/pull/10))
and a minimal `VPC` ([#16](https://github.com/alchemy-run/alchemy-effect/pull/16)).

```typescript
// roughly what you could deploy as of v0.2.0
export default Alchemy.Stack(
  "Hello",
  { providers: Cloudflare.providers(), state: Cloudflare.state() },
  Effect.gen(function* () {
    const bucket = yield* Cloudflare.R2Bucket("Photos");
    const kv     = yield* Cloudflare.KVNamespace("Cache");
    const site   = yield* Cloudflare.Worker("Site", {
      main: "./src/worker.ts",
      bindings: { PHOTOS: bucket, CACHE: kv },
    });
    return { url: site.url.as<string>() };
  }),
);
```

The other addition that turned out to matter more than it
looked was the first version of `Test/Bun`
([#15](https://github.com/alchemy-run/alchemy-effect/pull/15)) — a
`deploy(Stack) / destroy(Stack)` test harness that has
since underpinned every smoke test in the repo.

And **diff bindings** ([#12](https://github.com/alchemy-run/alchemy-effect/pull/12))
arrived alongside the first Queue Event Source — the
moment where "this resource depends on that one" stopped
being a static graph and started being something the
plan could reconcile against the live cloud.

## v0.3.0 — `Input<T>`, `Output<T>`, and a CLI

The two `v0.3.0` PRs are probably the most consequential
type-system changes in the project's history.

**`Inputs` and `Outputs`** ([#19](https://github.com/alchemy-run/alchemy-effect/pull/19)).
Before this release, resource props had to be concrete
values — you couldn't pass `bucket.bucketArn` directly
into another resource because it didn't exist yet at
declaration time. `Input<T>` boxed unresolved values so
the planner could resolve them lazily; `Output<T>` was
the dual on the producer side. Every cross-resource
reference in alchemy goes through this pair now.

**`DefineStack`, `defineStages`, and the `alchemy` CLI**
([#21](https://github.com/alchemy-run/alchemy-effect/pull/21)). The first
real CLI surface — `alchemy deploy`, `alchemy destroy`,
`alchemy plan` — and the stack-definition API that wraps
your `Effect.gen` body in a stage-aware `Alchemy.Stack`.

```sh
alchemy deploy --stage staging
alchemy destroy --stage pr-147
```

## v0.4.0 — Routing tables and Internet Gateways

The first networking primitives beyond `VPC`:
`InternetGateway`, `Route`, `RouteTable`, and
`RouteTableAssociation` ([#23](https://github.com/alchemy-run/alchemy-effect/pull/23)).
Nothing exotic, but routing a private subnet through an
IGW is the difference between "I can deploy a VPC" and
"I can deploy something *into* a VPC."

```typescript
const vpc  = yield* EC2.Vpc("App",        { cidrBlock: "10.0.0.0/16" });
const igw  = yield* EC2.InternetGateway("AppIGW", { vpc });
const rt   = yield* EC2.RouteTable("Public", { vpc });
yield* EC2.Route("PublicDefault", {
  routeTable: rt,
  destinationCidr: "0.0.0.0/0",
  internetGateway: igw,
});
```

## v0.5.0 — Standardize the names

`v0.5.0` was a quiet release that mostly fixed an
inconsistency you'd hit the second you used more than
one resource: physical names had no single source of
truth, and the field names varied by resource
(`worker.id`/`worker.name`, `bucket.name`, etc.).

Two changes:

- **`createPhysicalName(id)`** standardized — every
  resource that auto-generates a physical name now
  derives it from `${app}-${stage}-${id}`, truncated and
  branded with the same internal tags.
- **Renames**: `Worker.id` → `workerId`,
  `Worker.name` → `workerName`, `Bucket.name` →
  `bucketName` (with the AWS-side 3–63 character
  constraint enforced at the type level).

Boring in isolation, but it set the pattern every
resource added since has followed.

## v0.6.0 — NAT Gateway, EIP, SecurityGroups

The networking primitives that finished off a "real" VPC
stack: `NatGateway`, `EIP`, `EgressOnlyInternetGateway`,
and `SecurityGroup` ([#24](https://github.com/alchemy-run/alchemy-effect/pull/24)).

```typescript
const vpc = yield* EC2.Vpc("App", { cidrBlock: "10.0.0.0/16" });

// Public subnet via IGW (from v0.4) …
// Private subnet egresses through a NAT in the public subnet:
const eip = yield* EC2.EIP("NatEip");
const nat = yield* EC2.NatGateway("Nat", {
  subnet: publicSubnet,
  allocation: eip,
});

const appSg = yield* EC2.SecurityGroup("App", {
  vpc,
  ingress: [{ protocol: "tcp", port: 443, cidr: "0.0.0.0/0" }],
});
```

NAT Gateways turn out to be the resource where the
engine's retry-on-dependency-violation logic earns its
keep — they take ~90s to provision, and the EIP can't be
detached until the NAT is fully deleted. Both directions
need eventual-consistency handling, and `v0.6.0` is where
that pattern (catch `DependencyViolation`, sleep with
exponential backoff, retry) became the standard shape
for an AWS provider's delete path.

## Looking back

By the end of `v0.6.0`, alchemy was a working
Infrastructure-as-Effects engine with:

- a typed `Plan` / `Apply` core
- `Input<T>` / `Output<T>` for cross-resource references
- a `Binding` model with `Binding.Service` (runtime) +
  `Binding.Policy` (deploy-time)
- a CLI (`alchemy deploy/destroy/plan`)
- 4 Cloudflare resources, 1 AWS data resource, and
  enough of EC2 to deploy a real network

What was missing — and what `v0.6.1` was about to drop
in a single release — was *the rest of AWS*.
