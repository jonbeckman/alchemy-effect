The single most important commit in this range is the
one that replaced `create` + `update` with `reconcile`
([#179](https://github.com/alchemy-run/alchemy-effect/pull/179));
everything else in `beta.31` was tested against it before
shipping.
:::

`beta.21` through `beta.34` is the arc where the engine
went from "two-phase create / update lifecycle that
mostly works" to "one observe-and-converge `reconcile`
function that converges from any starting point." It's
also when `alchemy dev` for Cloudflare Workers shipped,
when `--adopt` made taking ownership of an existing
cloud resource a first-class flow, and when Neon,
AiGateway, Hyperdrive, and Tunnel/VpcService rounded out
the Cloudflare provider catalogue.

This is the release range where the project's internal
[reconciler doctrine](https://github.com/alchemy-run/alchemy-effect/blob/main/AGENTS.md#reconciler-doctrine)
went from "an idea" to "the rule every provider follows."

## `beta.21` — `pr-package`

**`@alchemy.run/pr-package`** ([#110](https://github.com/alchemy-run/alchemy-effect/pull/110))
from **Michael K** — the missing piece for using
alchemy across two repos. A PR build now publishes its
package to a per-PR npm tag (`alchemy@pr-147`); a
downstream consumer can `bun add alchemy@pr-147` and
test against the unreleased changes without waiting for
them to merge to `main`. The tag is pruned when the PR
closes.

The internal API is `pr-package` as a service that
hooks into GitHub Actions; the user-visible UX is a
comment on every PR with the install command.

Plus a `@distilled.cloud` bump to `0.13.1`.

## `beta.22` — D1 migrations + import + clone

**`Cloudflare.D1Database`** picks up three flags
([#111](https://github.com/alchemy-run/alchemy-effect/pull/111))
that turn it from "D1 wrapper" into "D1 with the dev
workflow people actually use":

```typescript
const db = yield* Cloudflare.D1Database("App", {
  migrationsDir: "./drizzle",          // apply migrations on deploy
  importFiles:   ["./seed.sql"],       // seed data on first deploy
  clone:         { from: ProdDb },     // clone a prod snapshot into this stage
});
```

`migrationsDir` watches a directory of `*.sql` files and
applies anything new on each deploy. `importFiles`
imports plain `.sql` files into the database (typically
seed data). `clone` is the powerful one — it takes a
snapshot from a source D1 database and clones it into
the target, which is how PR-preview stages can get a
copy of production data without re-seeding from scratch.

## `beta.24` — `alchemy dev` + `--adopt`

The big one. Two features that on their own would each
be a release headline:

### `alchemy dev` for Cloudflare Workers

**John Royal** ([#88](https://github.com/alchemy-run/alchemy-effect/pull/88))
landed the first version of `alchemy dev` for
Cloudflare. It's a local sidecar that:

- runs your Workers in `workerd` (the real Worker
  runtime, not a Node shim)
- proxies fetches between Workers running on the same
  host (so service bindings work locally)
- hot-reloads the Worker bundle on source change
- routes `*.localhost` hostnames to local Workers
  automatically (the polish for this landed in
  `beta.36`)

```sh
bun alchemy dev
# Workers running:
#   api      → http://api.localhost:8787
#   web      → http://web.localhost:8787 (proxies to api via env.API.fetch)
```

The dev story has been built out incrementally since —
Hyperdrive in dev mode (`beta.31`), `*.localhost`
resolution (`beta.36`), WASM bundling (`beta.37`), Vite
dev (`beta.40`) — but the original `beta.24` cut is
where the design constraints (real `workerd`, real
bindings, no Node shim) got set.

### `--adopt`

`alchemy deploy --adopt` takes ownership of cloud
resources that already exist. Before this, a name
collision (an S3 bucket exists; alchemy is asked to
create one with the same name) failed the deploy with
`ResourceAlreadyExists`. With `--adopt`, the engine
calls the provider's `read` hook, picks up the existing
resource's attributes, and treats it as if alchemy had
created it from the start.

```sh
alchemy deploy --adopt   # take ownership of pre-existing resources by name
```

`--adopt` doesn't take destructive action on the
existing resource — it just records that alchemy now
owns it, brands it with internal tags, and reconciles
the rest of the configuration on the next pass. The
underlying engine support (`read` returning `Unowned(attrs)`
to gate adoption) didn't fully land until `beta.31`'s
`reconcile` refactor; the `beta.24` shape was a
narrower per-provider opt-in.

Plus: `alchemy bootstrap` for Cloudflare — a single
command that provisions the state store and secrets
store and stashes the resulting tokens in the local
profile.

## `beta.25` — `keepAssets` removed

A short release that yanked the `keepAssets` Worker
optimization. It was meant to skip re-uploading
unchanged static assets between deploys, but its asset
hash was path-dependent — moving a file in the source
tree invalidated the entire upload. Removed; the
path-independent rewrite landed in `beta.31` ([#165](https://github.com/alchemy-run/alchemy-effect/pull/165)).

## `beta.26` — test alignment, raw Resource refs

Two structural fixes:

- **Vitest and Bun test helpers aligned** ([#129](https://github.com/alchemy-run/alchemy-effect/pull/129))
  — same shape (`test`, `beforeAll`, `afterAll`,
  `deploy`, `destroy`) regardless of which runner, plus
  profile integration so `bun test` picks up the active
  alchemy profile automatically.
- **Raw Resource refs as upstream dependencies**
  ([#122](https://github.com/alchemy-run/alchemy-effect/pull/122),
  from **Mathieu Post**). Previously, if you used a
  Resource as a value (e.g. a `Bucket` in a dependency
  list) without going through `Output<T>`, the planner
  didn't pick up the dependency edge. Now it does.

Plus the [State store guide](/concepts/state-store) was
written, types now ship as `lib/*.d.ts`
([#121](https://github.com/alchemy-run/alchemy-effect/pull/121)),
the CLI bin resolution went via package name
([#128](https://github.com/alchemy-run/alchemy-effect/pull/128)),
`alchemy state tree` parallelizes its reads
([#126](https://github.com/alchemy-run/alchemy-effect/pull/126)),
and a couple of bootstrap regressions in `SecretsStore`
and `StateStore` got fixed
([#131](https://github.com/alchemy-run/alchemy-effect/pull/131)).

## `beta.27` — CLI bundler hygiene

The CLI was getting bundled with `tsdown`, which was
rewriting `import.meta.path` in ways that broke the
generated Worker entrypoints. Stopped bundling; the
CLI ships as plain `.js` now
([#135](https://github.com/alchemy-run/alchemy-effect/pull/135)).

The `pr-package` install path also moved from `bun add
<name>` to `bun add <name>@<url>` to dodge a Bun
`DependencyLoop` bug
([#132](https://github.com/alchemy-run/alchemy-effect/pull/132)).

## `beta.28` — `PlatformServices` barrel cleanup

`PlatformServices` was being re-exported from `Util`,
which dragged the Node-specific bits into every Bun
import path. Pulled out of the barrel
([#137](https://github.com/alchemy-run/alchemy-effect/pull/137)).

## `beta.29` — AiGateway, version-gated state store, mobile polish

  inconsistent intermediate states.
- **Edge preview workers + Cloudflare Access**
  ([#155](https://github.com/alchemy-run/alchemy-effect/pull/155),
  **John Royal**) — work behind Cloudflare Access

- **Handle failed resource gracefully**
  ([#119](https://github.com/alchemy-run/alchemy-effect/pull/119))
  — a failed resource no longer cascades to mark the
  whole stack as failed; downstream resources that

- Website mobile polish from **Michael K**
  ([#146](https://github.com/alchemy-run/alchemy-effect/pull/146)):

## `beta.31` — the reconciler release

This is the biggest single release in the `2.0.0-beta`
line. There are six independent stories interleaved:

### `reconcile` replaces `create` + `update`

[#179](https://github.com/alchemy-run/alchemy-effect/pull/179).
Provider lifecycles used to be `create(news)` and
`update(news, olds, output)` — two functions, two code
paths, two surfaces for "what if this resource was
adopted, or had a stale state, or crashed halfway
through a previous run?"

`reconcile` collapses them into one. The provider
receives `output: Attributes | undefined` and
`olds: Props | undefined`, and the same body has to
work for all three combinations:

| `output`     | `olds`       | Meaning                         |
| ------------ | ------------ | ------------------------------- |
| `undefined`  | `undefined`  | Greenfield — first deploy        |
| defined      | defined      | Routine update                  |
| defined      | `undefined`  | Adoption (from `read`)          |

The required shape is documented as the
[reconciler doctrine](https://github.com/alchemy-run/alchemy-effect/blob/main/AGENTS.md#reconciler-doctrine):

1. **Observe** — derive the physical identifier; read
   live cloud state via `getX`/`describeX`.
2. **Ensure** — if missing, `createX`. Catch
   `AlreadyExists` as a race, continue.
3. **Sync** — for each mutable aspect, diff observed
   against desired, apply only the delta. Skip the API
   entirely on no-op.
4. **Return** — re-read if needed; return fresh
   `Attributes`.

The key shift: **observation > assumption**. Cloud
state is authoritative; `olds` is at most a hint to
skip a no-op call.

### Adoption via `read`

[#167](https://github.com/alchemy-run/alchemy-effect/pull/167).
With `reconcile` in place, `--adopt` becomes
mechanical: the engine calls `read(id)`, the provider
returns `Unowned(attrs)` if the resource exists but
lacks our ownership tags, and the engine gates
adoption behind the `--adopt` flag. Then `reconcile`
runs with `output` defined and `olds` undefined — the
adoption combination.

### Neon serverless Postgres

[#134](https://github.com/alchemy-run/alchemy-effect/pull/134).
**`Neon.Project`** and **`Neon.Branch`** — Postgres
projects with on-demand branches. Each `Neon.Branch`
exposes an `origin` output (pre-parsed connection
info) that `Cloudflare.Hyperdrive` accepts directly.
The shared-database pattern that `beta.37`'s
cross-stack references showcase was built on top of
this provider.

### Hyperdrive (with dev-mode local DB)

[#158](https://github.com/alchemy-run/alchemy-effect/pull/158),
**Julian Archila**. The connection-pooling layer for
Cloudflare Workers to Postgres. The dev-mode variant
runs a local Postgres in a container and routes the
Worker's Hyperdrive connection at it
([#242](https://github.com/alchemy-run/alchemy-effect/pull/242),
**John Royal** + **Sam**), so `alchemy dev` doesn't
require a real Neon project — your Worker talks to
local Postgres with the same `Hyperdrive` binding.

### Tunnel + VpcService + VpcServiceRef

[#123](https://github.com/alchemy-run/alchemy-effect/pull/123),
**Kotkoroid** + **Sam**. Cloudflare Tunnel + VPC
Service resources — connect a Worker to a VPC service
running anywhere reachable over a Cloudflare Tunnel
(self-hosted server, on-prem DB, etc.) without
opening it to the public internet.

### Workers + state-store observability

OTLP traces, metrics, and logs for the state store
worker ([#181](https://github.com/alchemy-run/alchemy-effect/pull/181))
— every state-store call is a span; failures become
metrics; structured logs go to Axiom (or any OTLP
endpoint).

Workers observability traces
([#244](https://github.com/alchemy-run/alchemy-effect/pull/244),
**Michael K**) — the Worker runtime emits the same
trace shape, so a request that hits a Worker then
calls the state store correlates end-to-end.

### AWS API Gateway hardening

[#169](https://github.com/alchemy-run/alchemy-effect/pull/169),
**Sam** + **Adrian Witaszak**. `ApiGateway.RestApi`,
`Resource`, `Method`, `Integration`, `Stage`,
`Deployment` got proper retry + binding semantics,
including stage-level patching for tracing, throttling,
and access-log configuration. Tested against the
reference reconciler in S3 / SQS / Kinesis.

## Looking back

By `beta.34`:

- The engine had a single, doctrine-bound `reconcile`
  function on every provider.
- `--adopt` was wired through the engine via `read`.
- `alchemy dev` ran Cloudflare Workers locally with
  real bindings (and a local Postgres for Hyperdrive).
- Neon, AiGateway, Hyperdrive, Tunnel/VpcService were
  all first-class.
- The CF state store had OTLP observability and
  version-gating; the CF state worker would refuse to
  serve a stale schema.
- The PR-package workflow was working end-to-end.

The next release, `beta.35`, is the first one with a
proper release blog post — and from there the cadence
of "release + blog" became the default. This is where
the backfill stops and the original narrative picks up:

- [`alchemy@2.0.0-beta.35`](/blog/2026-05-08-beta-35) —
  the Effect-native queue consumer, R2 custom domains,
  non-string Worker env bindings, Neon logical
  replication.
